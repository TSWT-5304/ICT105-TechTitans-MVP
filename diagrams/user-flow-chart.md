## Case Study Scenario

**PeerTutor**, an online student-to-student tutoring platform that connects students with qualified peer tutors. 
- Students can register or log in, search for tutors, view tutor profiles, submit tutoring requests, confirm tutoring sessions, and leave reviews after completed sessions.
- Students who wish to become tutors must first submit a tutor registration application for admin approval. Once approved, tutors can create their tutor profiles, receive and manage tutoring requests, and view reviews from students.
- Administrators oversee the platform by managing users, reviewing tutor applications, and monitoring platform activities through an administrative dashboard.

## User Flow-Chart Description

1. The user opens the **PeerTutor homepage**.
2. The user selects a role (**Student, Tutor, or Admin**).
3. If the user does not have an account, they **register**; otherwise, they **log in**.
4. Students search for tutors, browse tutor listings, and view tutor profiles.
5. If a suitable tutor is found, the student submits a **Tutoring Request Form**.
6. Tutors submit the **Tutor Registration Form**, and the admin reviews the application.
7. If the tutor application is **rejected**, the tutor process **ends**. If **approved**, the tutor creates a tutor profile and waits for student requests.
8. Tutors review incoming tutoring requests and either **approve** or **reject** them.
9. If the tutoring request is **rejected**, the student returns to search for another tutor. If **approved**, the student confirms the tutoring session.
10. The tutoring session takes place.
11. After the session, the student leaves **reviews and ratings**, and the **student process ends**.
12. The tutor views the student's reviews and ratings, then returns to **waiting for new student requests**, repeating the process for future tutoring sessions.
13. Throughout the process, the admin manages users, monitors tutor applications, and views the summary dashboard. The **admin process ends when the admin logs out**.
